# 合并上游 v0.7.1 方案

## 一、目标

将上游仓库 `ZyphrZero/kiro.rs` 的 `v0.7.1` 功能融合到当前 `kiro.rs.huanchun`，同时保留本项目已有的多端点路由、响应缓存、Prompt 过滤、计费治理、TLS 指纹、管理端和客户端密钥体系。

本次融合以当前项目代码为主线，按功能手工移植上游修改，不直接覆盖本地源码。

## 二、零回归硬性要求

本次融合的第一优先级是保证当前本地版本的全部功能、配置和数据格式不受影响。上游 `v0.7.1` 只能作为新增功能和缺陷修复的来源，不能改变本地已经稳定运行的行为。

必须遵守以下规则：

- 当前 `master` 是唯一功能基线，上游代码不能取代本地主线。
- 只允许按函数或功能点手工移植，禁止使用上游文件整体覆盖本地文件。
- 禁止删除本地已有路由、配置字段、环境变量、管理端入口、数据库字段和凭据字段。
- 禁止改变现有 `/v1/messages`、`/cc/v1/messages`、`/v1/chat/completions` 的请求或响应格式。
- 禁止改变现有客户端 `csk_*`、主 API Key、分组鉴权和用量统计行为。
- 禁止改变现有凭据文件、配置文件、统计文件和 SQLite 数据库的兼容性。
- 禁止削弱多端点回退、并发调度、账号冷却、响应缓存、Prompt 过滤、计费治理、请求追踪和 TLS 指纹功能。
- 新增 `/v1/responses` 必须是独立增量入口，不能通过替换现有 Chat Completions 实现来接入。
- 每个阶段必须独立提交、独立测试；任意现有功能测试失败，该阶段不得进入下一阶段。
- 未完成全部回归测试和真实环境冒烟测试前，融合分支不得合并回 `master`，也不得用于正式环境。

“保证不受影响”的验收定义如下：

1. 当前本地版本已有测试在融合前后全部通过。
2. 为当前本地定制功能补齐专项回归测试，融合后全部通过。
3. 使用同一份正式配置和脱敏后的同结构凭据进行启动验证，无需迁移即可正常加载。
4. 现有调用方无需修改 URL、请求体、API Key 或解析逻辑。
5. 管理端页面、批量导入、凭据管理、分组、统计、日志和在线更新入口均可正常使用。
6. 性能关键路径没有明显退化，启动时间、普通对话延迟和流式首包行为保持在可接受范围内。
7. 出现任何回归时，可以按独立提交立即回滚到当前 `57409c1` 的行为。

## 五、需要融合的内容

### 阶段一：低风险基础修复

每一项必须形成独立 Git 提交，确保可以单独回滚。

#### 1. Admin 请求体限制

参考上游提交：`3170a64`

在 `src/admin/router.rs` 增加 50MB 的 `DefaultBodyLimit`，修复大量凭据批量导入时超过 Axum 默认 2MB 限制而返回 HTTP 413 的问题。

不得改动现有公开 OAuth 回调路由和管理端认证中间件。

#### 2. 凭据原子落盘

参考上游提交：`e0e77c7`

修改本地 `src/kiro/token_manager/mod.rs` 中的 `persist_credentials`：

- `persist_lock` 必须覆盖快照、序列化和写盘整个临界区。
- 获取写入锁后再从内存重新生成最新凭据快照。
- 先写同目录临时文件，再通过 `rename` 原子替换正式文件。
- 写入失败时清理临时文件。
- 保留本地凭据规范化、禁用状态和禁用原因持久化逻辑。

#### 3. 上游 429 完整传播

参考上游提交：

- `9230934`
- `326a2c7`

需要增加类型化的 `UpstreamRateLimitError`，并贯通 Provider、Token Manager、WebSearch、MCP 和 HTTP 响应映射：

- 上游最终返回 429 时，客户端也必须收到 429。
- 合法的 `Retry-After` 必须原样保留。
- 非法 `Retry-After` 不应写入响应头。
- Token 刷新遇到 429 时不能把凭据永久禁用。
- 分组请求只能统计当前分组内的剩余凭据。
- 普通上游错误不能向客户端泄露账号、请求 ID 或原始响应体。

本地现有账号冷却、重试退避、并发调度和熔断逻辑必须保留。

#### 4. Enterprise 余额及模型查询

参考上游提交：`ccdcffd`

调整 `getUsageLimits` 和 `ListAvailableModels`：

- 请求 URL 不再携带 `profileArn` 查询参数。
- 保留本地多区域候选和端点回退策略。
- 保留流式请求对真实 `profileArn` 的解析和注入。
- `profileArn` 仍可用于确定数据面区域，但不能附加到这两个旧版 REST 查询 URL。
- `setUserPreference` 和流式接口不跟随本项盲目修改，应按各自上游协议处理。

### 阶段二：Tool Call 全链路加固

参考上游提交：

- `b304466`
- `31aefbe`

本地 Schema 规范化已经覆盖 `31aefbe` 的主要功能，禁止直接覆盖整个 `converter.rs`。重点补充以下缺口：

- 引入按 `tool_use_id` 缓冲的 `ToolJsonAccumulator`。
- 工具参数只有在收到 stop 后才整体解析。
- 半截或非法 JSON 不再静默回退为 `{}`。
- 非流式请求应返回明确的 502。
- 流式请求应输出合法的 error 事件。
- 识别 `TOOL_SCHEMA_INVALID` 等客户端 Schema 错误。
- 增加 `toolCompatibilityMode`，支持 Claude Code 内置工具名称及参数双向转换。
- 过滤跨 chunk 的 `<tool_use>` XML 泄漏。
- 统一流式、非流式和 WebSearch 的工具名称恢复逻辑。

必须保留本地现有的 `<invoke>` 恢复、严格工具邻接清理、Prompt 过滤和响应缓存策略。

### 阶段三：新增 `/v1/responses`

参考上游提交：

- `79f8f10`
- `3b086ee`
- `232e727`

上游实现位于 `src/anthropic/responses.rs`，本地应按照现有模块结构移植到 `src/openai/responses.rs`，不能覆盖本地 `src/openai/`。

需要完成：

- 注册 `POST /v1/responses`。
- 复用本地 `AppState`、认证中间件、客户端分组、计费、追踪和 Provider。
- 支持 Responses 流式与非流式输出。
- 支持 `function` 工具。
- 支持 `custom` 工具。
- 支持 `namespace` 工具组及名称展平、还原。
- 支持 `additional_tools`。
- 支持 `function_call_output` 和 `custom_tool_call_output` 续接。
- Developer 消息转换为 System。
- 支持 reasoning 输出和相关 SSE 事件。
- 支持 `web_search_call` 输出和相关 SSE 事件。
- 工具结果之后遇到空响应时执行有限重试，最终仍为空时返回 502。

现有 `/v1/chat/completions` 必须继续使用本地实现，并执行完整回归测试。

### 阶段四：鉴权兼容性评估

上游提交 `6598531` 和 `b9fbda7` 取消了 API Key 前缀校验，并将自动生成的 Key 统一为 `sk-`。

本地管理端、UI、分组、计费及存量用户仍依赖 `csk_*`。本轮融合不直接应用这些提交，继续保持现有 Key 可用。

如以后迁移，应采用兼容策略：

- 现有 `csk_*` 永久保持可验证，不能强制失效。
- 新 Key 是否生成 `sk-` 单独决定。
- 配置文件中的主 API Key 继续按完整值精确匹配。
- 迁移需要独立提交和独立发布说明。

## 七、建议 Git 操作流程

开始实施前创建保护分支和标签：

```cmd
git switch master && git status && git tag backup-before-zyphrzero-v0.7.1 && git switch -c merge/zyphrzero-v0.7.1
```

开始改动前还必须保存以下基线信息：

- `cargo test --no-default-features` 的完整结果。
- 管理端生产构建结果。
- 当前所有 HTTP 路由清单。
- 当前配置结构和默认值快照。
- 当前凭据、客户端 Key、统计和追踪数据的加载测试结果。

- 多端点回退、账号分组、限流冷却、缓存、Prompt 过滤、计费和追踪的行为样本。

这些基线用于融合后的逐项对比。测试输入、配置和期望结果应固定，避免仅凭“能够启动”判断没有回归。

建议提交顺序：

1. `fix(admin): 管理端请求体限制提升至50MB`
2. `fix(token): 凭据文件原子落盘`
3. `fix(rate-limit): 完整传播上游429和Retry-After`
4. `fix(enterprise): 修复余额和模型查询参数`
5. `fix(tools): 加固工具调用JSON和错误处理`
6. `feat(responses): 增加OpenAI Responses基础接口`

8. `test: 增加v0.7.1融合回归测试`

不得把所有内容压成一个提交，也不应在验证前合并回 `master`。

## 八、验证要求

### Rust 基础验证

为避免默认 TLS 指纹功能触发 LLVM/libclang 构建依赖，先运行：

```cmd
cargo fmt --check && cargo test --no-default-features
```

已安装项目所需 LLVM/libclang 后，再验证默认功能或正式发布功能组合。

### 必须增加的回归测试

- Admin 批量导入超过 2MB 时不返回 413。
- 并发更新凭据不会发生旧快照覆盖新 Token。
- 凭据写入成功后 JSON 完整且没有临时文件残留。
- 普通 429、账号级 429、刷新阶段 429 均按预期处理。
- `Retry-After` 合法值保留，非法值丢弃。
- Enterprise 余额和模型 URL 不含 `profileArn`。
- 原有 `/v1/messages`、`/cc/v1/messages` 和 `/v1/chat/completions` 行为不退化。
- `/v1/responses` 流式和非流式返回结构正确。
- function、custom、namespace 工具声明、调用和结果续接正确。
- reasoning、web search 和工具调用 SSE 事件顺序正确。
- 空工具续接响应能够重试并在失败后返回 502。
- 客户端 Key 分组、用量统计、Prompt 过滤和请求追踪继续生效。

验证通过后，再将融合分支合并回 `master` 并推送到正式 GitHub 仓库。
